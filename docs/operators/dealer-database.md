# Dealer Database

Heat & Glo sells through ~1,400 dealers across the US + Canada. The dealer locator is one of the highest-conversion surfaces on the site. This ledger tracks how the dealer data is sourced, stored, refreshed, and surfaced.

---

## Current state (2026-05-01)

**Status:** dealer dataset is NOT YET populated in the rebuild. The legacy site's `/where-to-buy/` is JS-rendered, so the Phase 8 static crawl couldn't enumerate it. A Playwright-driven enumeration pass is queued.

**Source of truth (legacy):** `https://heatnglo.com/where-to-buy/` (JS widget), backed by HHT's internal dealer-management system. We don't have direct access to that backing system.

**Where it WILL live in the rebuild:** `src/data/dealers.json` (a static data file generated by the Playwright pass + Google Places enrichment, not a content collection). Not editable in CloudCannon — too many records and operators don't add dealers, HHT marketing does.

---

## Generation process (Playwright pass)

When the dealer data needs to be (re)generated:

1. Run `scripts/scrape-dealers.mjs` (TODO: write this script as part of the dealer-database task).
2. Script uses Playwright to browse the legacy locator, enumerate every dealer, capture name + address + phone + dealership tier (Platinum / Gold / Silver / authorized).
3. Script writes raw output to `docs/data/dealers-raw.json`.
4. Enrichment pass: hit Google Places API for each dealer to get place_id, lat/lng, hours, rating, review count.
5. Final output written to `src/data/dealers.json`. Schema:
   ```
   {
     "dealers": [
       {
         "id": "dealer-slug",
         "name": "Smith's Hearth & Patio",
         "address": "...",
         "city": "Charlotte", "state": "NC", "country": "US", "zip": "28202",
         "phone": "...",
         "tier": "platinum",
         "google_place_id": "ChIJ...",
         "lat": 35.227, "lng": -80.843,
         "rating": 4.8, "review_count": 127,
         "hours": { "mon": "9-6", ... }
       }
     ]
   }
   ```

## Refresh cadence

| Reason | Trigger | Frequency |
|---|---|---|
| HHT adds or removes dealers | Playwright re-pull + Places re-enrichment | Quarterly minimum |
| Dealer changes hours / phone | Places re-enrichment only | Monthly (cheaper) |
| Dealer reviews accumulate | Places re-enrichment only | Monthly |
| Major HHT distribution change | On-demand | As needed |

## How the dealer data is surfaced on the site

| Surface | What's shown | Data source |
|---|---|---|
| `/where-to-buy/` locator widget | Search by ZIP/postal code, see nearest dealers on a map + list | `src/data/dealers.json` filtered by lat/lng radius |
| Product page `Dealer CTA` block | "Find a dealer near you" + ZIP capture | Same data source |
| Homepage hero ZIP capture | Same widget on the hero | Same data source |
| Future: per-dealer landing pages | One page per dealer with their featured products | Same data source + a Phase 10 build |

Per the locked AU/NZ IA decision, AU/NZ product pages route to AU/NZ dealer endpoint (`?region=au-nz`), US pages route to US endpoint. Branched dealer routing is part of the AU/NZ hreflang implementation task.
